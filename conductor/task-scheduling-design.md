# 定时任务调度功能设计文档 (Task Scheduling Design)

## 1. 概述 (Overview)

本项目旨在为统一云盘自动转存系统 (UCAS) 引入定时任务调度能力。用户可以在创建或编辑任务时配置 Cron
表达式，系统将在后台自动、定时地触发转存逻辑。设计目标是轻量、高效、安全，并提供良好的用户交互体验。

## 2. 架构设计 (Architecture)

采用**热插拔式独立调度器 (Hot-Pluggable Independent Scheduler)** 架构：

- **核心依赖**: 引入成熟的开源库 `github.com/robfig/cron/v3` 作为底层的定时触发引擎。
- **模块封装**: 在 `internal/core` 下新建 `scheduler` 模块，负责管理 Cron
  实例的生命周期（启动、停止、添加任务、移除任务）。
- **无缝集成**: 调度器与现有的 `worker.Manager` (执行引擎) 和 `db` (存储)
  解耦。调度器仅负责“到点触发”，触发后将任务封装为 `worker.Job` 提交给现有的 Worker 队列执行。

## 3. 核心组件与数据流 (Components & Data Flow)

### 3.1 后端调度器 (`internal/core/scheduler/scheduler.go`)

- **初始化与加载**: 系统启动时 (`main.go`)，实例化 `Scheduler` 并从数据库读取所有 `cron` 字段不为空且状态不是
  `[Fatal]` 的任务，注册到 Cron 引擎中。
- **动态同步**: 在 API 层 (`router.go`)，当任务被创建、更新或删除时，调用 Scheduler 的暴露方法（如
  `UpdateTask(taskID)` 或 `RemoveTask(taskID)`），保持内存调度与数据库状态实时一致。
- **触发逻辑 (安全保护)**: 当 Cron 规则触发时，执行的闭包函数首先从数据库重新拉取该任务的最新状态：
  - 如果状态为 `running`，则**静默跳过**本次执行，防止多个相同任务重叠执行导致网盘 API 封控。
  - 如果状态为 `failed` 且包含 `[Fatal]` 标记，则跳过执行（致命错误需人工干预）。
  - 其他状态下，正常提交给 Worker 队列，并将状态更新为 `running`。

### 3.2 前端交互 (`web/src/views/Tasks.vue`)

- **定时开关**: 在任务编辑表单中增加一个“启用定时执行”的 Switch 开关。
- **可视化选择器**: 开启开关后，展示一个 Select 下拉框，提供常用预设（如“每天凌晨2点”、“每6小时”、“每周一”等对应的内部 Cron
  字符串）。
- **自定义模式**: 下拉框中保留一个“自定义”选项。选择后，显示一个 Input 文本框，允许进阶用户直接输入标准的 5/6 位 Cron
  表达式，并配有基础的格式提示。

### 3.3 数据库模型 (`internal/db/db.go`)

- 复用现有的 `Task` 模型中的 `Cron` 字段（`string` 类型）来存储标准 Cron 表达式。
- `NextRun` 字段由业务逻辑或前端展示时可选计算，核心调度依赖 Cron 引擎自身的内部时钟。

## 4. 错误处理与容错 (Error Handling)

- **表达式校验**: 前端和后端 API 均需对自定义输入的 Cron 表达式进行基础校验，防止无效的表达式导致调度器 panic 或任务注册失败。
- **重启恢复**: 系统重启后，能够根据数据库中的 `cron` 字段自动恢复所有激活的定时任务，无需人工干预。

## 5. 测试策略 (Testing)

- **单元测试**: 针对 `scheduler` 模块的注册、移除逻辑进行测试，模拟任务状态变化。
- **集成验证**: 验证 API 层更新任务后，底层的 Cron 触发时间是否按预期改变。
- **并发保护测试**: 模拟任务执行耗时大于 Cron 周期，验证是否能正确跳过重叠的执行。
