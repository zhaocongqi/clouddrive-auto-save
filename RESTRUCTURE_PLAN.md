# 统一云盘自动转存系统重构与合并计划 (Unified Cloud Auto-Save)

## 1. 背景与动机 (Background & Motivation)

目前工作区中存在两个针对不同网盘的自动化转存项目：

* **`cloudpan-auto-save`**: 基于 Node.js，专注于**移动云盘 (139)**。
* **`quark-auto-save`**: 基于 Python，专注于**夸克网盘 (Quark)**，拥有强大的正则重命名和插件系统。

维护两个不同技术栈（Node.js / Python）、不同 UI 和不同后台调度逻辑的项目成本较高，且用户的操作体验割裂。
本计划旨在**重构代码、设计全新 UI 并整合两者的核心能力**，构建一个针对 139 和 Quark 的高性能、低内存占用、单体化部署的全新云盘自动化转存系统。

## 2. 范围与预期影响 (Scope & Impact)

* **核心支持**：深度融合 **移动云盘 (139)** 和 **夸克网盘 (Quark)** 的底层 API 交互。
* **高级调度与处理**：整合定时任务引擎、自动重试机制、跨平台的文件重命名规则（合并 Jinja2 模板与夸克的正则/魔法变量）。
* **全新可视化 UI**：抛弃旧版简易 HTML 与 Python WebUI，采用现代化前端框架实现响应式、暗黑模式支持的统一控制台。
* **强大的机器人与通知**：统一整合 Telegram 机器人命令、系统日志 SSE 流、以及多渠道消息推送（Bark, WxPusher）、Emby/Alist 联动。
* **预期影响**：为用户提供一个无需依赖复杂运行环境的单执行文件（或极轻量 Docker 镜像），大幅降低服务器资源占用，提升稳定性和并发处理能力。

## 3. 技术方案 (Proposed Solution)

根据技术选型，确定了以下技术栈：

* **后端开发语言**: **Go (Golang)**。利用 Go 的高并发特性（Goroutine）和低内存占用，非常适合处理大量的网盘 API 请求和本地文件 I/O，且能编译为跨平台的单体二进制文件。
* **前端框架**: **Vue 3 (Vite + Element Plus / Tailwind CSS)**。提供现代化的组件库和极佳的开发体验，构建的静态文件将被 Go 1.16+ 的 `embed` 特性内嵌，实现前后端一起分发。
* **数据库**: **SQLite (借助 GORM 或 Ent)**。保持轻量级，零配置，完美继承个人云工具的数据存储需求。

## 4. 阶段性实施计划 (Phased Implementation Plan)

### 第一阶段：项目初始化与底层设计 (✅ 已完成)

1. **Go 工程搭建**：初始化 Go Modules，确立项目结构（cmd, internal, web）。
2. **数据模型设计**：统一数据库表结构，基于 GORM 实现 Account, Task, Setting 模型。
3. **云盘接口抽象**：设计统一的 `CloudDrive` 接口，定义标准文件操作与转存契约。

### 第二阶段：云盘 SDK 的 Go 移植 (✅ 已完成)

1. **139 (移动云盘)**：移植登录逻辑、mcloud-sign 签名算法及全套 HCY 私有接口。
2. **Quark (夸克网盘)**：移植 Cookie 解析、双端（PC/App）请求模拟及异步转存逻辑。

### 第三阶段：任务调度与规则引擎重构 (✅ 已完成)

1. **调度器 (Scheduler)**：实现基于 Goroutine 的 Worker Pool，支持任务并发执行。
2. **双层 Cron 系统**：实现“全局开关+局部重写”的调度逻辑，支持秒级 Cron 表达式。
3. **重命名引擎 (Renamer)**：整合正则匹配与“魔法变量”，支持 Go Template 动态渲染。
4. **流程整合**：完成“转存+自动扫描+重命名”的完整任务闭环，支持重命名感知的同名跳过。

### 第四阶段：现代化 Web UI 开发 (✅ 已完成)

1. **项目搭建**：使用 Vite + Vue 3 + Element Plus 构建，引入 Lucide 图标库。
2. **界面实现**：完成 Dashboard 仪表盘（支持“已规划任务”统计）、侧边栏导航、暗黑模式切换。
3. **监控中心重构**：实现“实时执行状态”面板，支持百分比进度显示、高度对齐布局及等宽字体监控视图。
4. **状态一致性**：通过“清空日志”联动重置数据库任务摘要，确保各组件视图状态高度同步。
5. **功能页面**：实现账号管理（原地更新、校验耗时监控）与任务管理（列表、创建、手动运行、重命名感知选择）。
6. **错误处理系统**：实现针对 139/Quark 的人性化错误码映射与 `[Fatal]` 致命错误自动阻断/UI 警告机制。

### 第五阶段：集成生态与通知体系 (📅 待开始)

1. **Telegram Bot**：重新实现指令（如 `/tasks`, `/search_cs`）。
2. **推送通知**：实现 Bark, WxPusher, PushPlus 和 企业微信推送。

### 第六阶段：最终部署与数据迁移 (📅 待开始)

1. **静态资源内嵌**：利用 Go `embed` 嵌入前端页面。
2. **Docker 镜像构建**：编写 Multi-stage 跨平台 Dockerfile。

### 第七阶段：AI 智能化增强 (📅 待开始)

1. **AI 智能重命名**：接入 LLM（如 Gemini/OpenAI），通过自然语言指令完成复杂文件名整理，告别正则。
2. **AI Cron 助手**：前端集成自然语言生成 Cron 表达式功能，如“每周五下午两点”一键转换。

## 5. 验证与测试 (Verification & Testing) (✅ 已完成)

* 针对重命名引擎编写了详尽的单元测试，覆盖所有魔法变量。
* 使用 Mock 框架针对云盘 API 进行集成测试，解耦了数据库与驱动依赖。
* 在真实环境进行了高并发转存压力测试，验证了 Worker 池的稳定性。
